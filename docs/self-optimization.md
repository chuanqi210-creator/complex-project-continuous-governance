# Complex Self-Optimization

Complex optimizes itself through the existing behavior kernel, external-calibration discipline, evaluation records, and maturity registry. This is a maintenance mode, not another runtime mechanism.

Its objective is to improve real project outcomes while keeping the active protocol small. A maintenance beat is not complete because prose changed or a checker passed. It is complete only to the evidence level actually reached, with the next evidence debt either executed, queued, blocked by a real responsibility boundary, or explicitly retired.

## 1. Select The Change Path

| Change class | Typical scope | Required path |
| --- | --- | --- |
| Routine correction | typo, broken link, exact verifier repair, wording with no semantic default change | edit, test, residual scan |
| Bounded improvement | recurring behavior failure inside an existing mechanism or boundary | behavior case, layer diagnosis, reversible change unit, bounded evaluation |
| Substantial change | public term, schema, core default, Codex-surface contract, mechanism role, or maturity claim | compact change packet and progressive evidence rollout |

Do not promote a routine correction into governance ceremony. Do not hide a substantial behavior change inside a wording patch.

Framework Grilling is conditional. Use it only if inspection leaves a user-owned decision fork that can change the destination, architecture, responsibility boundary, or evaluation. Otherwise record `no_grill_needed` and keep maintenance design with the AI.

## 2. Run One Self-Optimization Cycle

1. **Restore the baseline.** Read the active architecture, mechanism maturity, current recovery state, related behavior cases, current external evidence, and the concrete failure or opportunity.
2. **Define the change unit.** Classify the change, name the failed Prompt/Context/Harness/Loop layer, and define one reversible candidate. Record its baseline, intended outcome, failure condition, compatibility impact, and rollback.
3. **Calibrate against implementation.** Reuse fresh pinned evidence or inspect a relevant implementation's goal/non-goals, state and control path, code/configuration, tests/evaluations, operating limits, and failure boundaries. Transfer one micro-contract, not a project name.
4. **Write the falsifiable contract first.** Add or refine the behavior case, transcript rule, outcome predicate, and grader boundary before editing the normative surface.
5. **Implement within the current architecture.** Prefer merging the candidate into the behavior kernel, a universal boundary, a supporting practice, or a conditional extension. Create a new mechanism only after repeated cross-project evidence shows that none of those homes is coherent.
6. **Run progressive evidence rollout.** Keep the stable baseline available while the candidate advances only as far as evidence permits.
7. **Decide explicitly.** Choose `promote`, `keep_candidate`, `demote`, or `rollback`. Update public wording, examples, references, maturity, and recovery state only to that level.
8. **Scan residual evidence debt.** Execute the next internally authorized validation while it remains useful and feasible. Stop only when the Goal-level completion predicate and residual scan pass, or a true blocker or responsibility boundary remains.

For a long maintenance program, the Prompt Contract also sets a time appetite, next stage result horizon, and usable stage artifact. By that horizon, deliver the verified change packet, bounded comparison, promotion decision, or decision-grade failure diagnosis promised for the stage. If evidence work does not fit, narrow the candidate or park lower-value evidence debt without lowering the graduation standard.

## 3. Progressive Evidence Rollout

This path adapts mature change-proposal and progressive-delivery systems without importing their platform machinery.

| Evidence boundary | Required observation | Permitted claim |
| --- | --- | --- |
| Contract screen | behavior case, transcript rule, external implementation boundary, rollback | candidate is specified and screenable |
| Bounded reproduction | fixed upstream fixture or local mechanism probe | the bounded mechanism can run under that fixture |
| Same-task comparison | locked baseline/candidate, one changed variable, outcome/trajectory/human graders | relative behavior on that task |
| Shadow or limited use | observable real-project scope, stable baseline, abort/rollback route | candidate is usable in the observed scope |
| Repeated outcome review | distinct real or redacted-real samples and independent human review | evidence may support maturity promotion in the stated scope |

These boundaries do not replace `screened`, `tested`, or `validated`. They describe how evidence is gathered. Maturity changes only through `docs/mechanism-maturity.json` promotion rules.

## 4. Compact Change Packet

A substantial change records only what another maintainer needs to judge and recover it:

- problem evidence and affected project outcomes;
- Goal and non-goals;
- baseline and reversible candidate;
- external implementation and transfer boundary;
- alternatives, drawbacks, compatibility, and operating cost;
- evaluation case, graduation criteria, failure criteria, and validity threats;
- rollout scope, observability, rollback, and owner;
- decision status and next validation.

The packet authorizes a candidate experiment. It does not validate the candidate.

## 5. Graduation And Stop Rules

- A behavior marker can screen wording; it cannot prove project value.
- An upstream success or local reproduction cannot validate Complex.
- A candidate does not replace the stable baseline without observable graduation and rollback conditions.
- A fixed number of beats is not a stop condition.
- If the residual evidence scan finds an executable validation that can change the decision, continue without asking the user for routine permission.
- Ask only when the next step crosses the responsibility boundary or needs unavailable external input.

The machine-readable sources of truth remain `docs/active-architecture-rebaseline.json`, `docs/mechanism-maturity.json`, `docs/reference-implementation-evidence.json`, the behavior pack, evaluation records, and `protocol/current-state.md`.
