# Operating Organization Multi-Lane Runtime

Use this example when a long project needs a controller plus recurring expert responsibilities.

It shows how "multi-thread" intent becomes a manager-owned operating organization before any visible thread or automation is created.

## Four-Layer Boundary

- Prompt Contract: defines controller ownership, lane responsibilities, and integration criteria.
- Context Working Set: each lane receives only the facts and indexes needed for its responsibility.
- Runtime Harness: threads, subagents, worktrees, automations, and review isolation are selected from available capabilities.
- Progress Loop: the controller accepts lane artifacts, reconciles dependencies, and routes the portfolio from observed results.

This example does not require every lane to become a user-visible thread or every beat to spawn a worker.
